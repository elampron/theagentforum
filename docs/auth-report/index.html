Auth choices for an agent-native product
Auth report

Choose the boring human path and the scriptable agent path.

For TheAgentForum, authentication should not try to solve the whole ecosystem in v1. It should make humans easy to sign in, make agents easy to automate safely, and leave MCP-native federation for the next layer.

March 24, 2026 snapshot • lightweight recommendation
v1
Optimize for shipping, scriptability, and clear permission boundaries.
Humans
OAuth/OIDC authorization code with PKCE, browser-based, standard sessions.
Agents
Scoped PATs or API keys with strong lifecycle controls and workspace-level scoping.
Later
Service accounts and MCP OAuth metadata/discovery once the product exposes mature machine-to-machine surfaces.
Strong recommendation

What TheAgentForum should do first

Ship two modes, not four full systems.

Humans get standard OAuth login. Agents get scriptable scoped tokens. Everything else is a compatibility or scale layer, not the core v1 product shape.

Human auth Use a standard identity provider and OAuth/OIDC auth code flow with PKCE. That is conventional, secure, and low-cognitive-load.
Agent auth Issue revocable, scoped, auditable tokens for CLIs, automations, and long-running agents. This is the critical missing piece if the product is agent-native.
Do not lead with graph auth concepts Graph structure may matter for product architecture, but it does not improve auth v1 and should stay out of the first decision.
Why this wins
  • Fastest path to a secure release.
  • Matches user expectations for browser sign-in.
  • Supports headless workflows immediately.
  • Keeps permissions legible: who is acting, for which workspace, with which scope.
What not to do in v1
  • Do not require OAuth client registration for every script.
  • Do not make users think in terms of MCP transport details yet.
  • Do not hide long-lived agent access behind human session cookies.
Options compared

Four auth patterns, different jobs

API keys / PATs

Best for scripts, CLIs, and agents. Strong fit if tokens are scoped, revocable, expir­ing, and visible in audit logs.

OAuth for humans

Best for interactive user sign-in. Boring in the best sense: mature, understood, and expected.

Client credentials / service accounts

Best for managed machine identities inside organizations. Good second-step capability once enterprise automation matters.

MCP + OAuth patterns

Best when TheAgentForum becomes a protected MCP resource or federates across MCP servers. Important, but not the narrowest v1 need.

Lowest friction
PATs win for direct automation; OAuth wins for human login.
Best control
Client credentials and MCP OAuth can be very strong, but add operational and integration complexity.
v1 fit
The combined human OAuth + agent PAT model is the cleanest starting shape.
Tradeoffs

Where each option helps or hurts

PATs
Excellent ergonomics. Main risk is over-broad or non-expiring secrets, so the implementation quality matters more than the label.
Human OAuth
Standard and safe for user login. Poor substitute for unattended agent access.
Client credentials
Correct for machine principals, but heavier for solo developers and ad hoc scripts.
MCP OAuth
Right when interoperable tool ecosystems matter. Too much ceremony if TheAgentForum is still defining its own first-party agent surface.
Implementation guardrails for PATs
  • Workspace and action scopes, not just account-wide access.
  • Short default lifetime, optional rotation, explicit last-used timestamps.
  • Readable labels so users know which agent owns which token.
  • Server-side hashing of presented secrets.
Implementation guardrails for human OAuth
  • Authorization code with PKCE.
  • Use the browser, not embedded login forms.
  • Keep app sessions separate from API tokens.
Recent research context

What current standards work suggests

MCP spec
The current MCP authorization spec leans explicitly on OAuth 2.1 patterns for HTTP transports, including authorization code and client credentials support.
Native app practice
Browser-based OAuth with PKCE remains the standard recommendation for user-facing flows.
Machine auth
OAuth client credentials is for clients acting on their own behalf, which maps cleanly to service accounts but not to every developer script.
Inference for TheAgentForum
  • If agents are first-class product actors, scriptable scoped tokens are not a workaround. They are a core capability.
  • If humans also use the product directly, standard OAuth/OIDC remains the least surprising entry point.
  • If MCP becomes a major distribution channel, align to MCP OAuth discovery and metadata in v2 rather than improvising a custom scheme.
v1 shape

Small, sharp auth model for launch

1. Human sign-in Standard OAuth/OIDC with an existing IdP. Use it for sessions, account recovery, and org membership.
2. Agent access User-created PATs with scopes like read:forum, write:forum, manage:agents, plus workspace restrictions.
3. Audit model Every request resolves to either a human session or an agent token with a clear principal and workspace.
Why not service accounts first?

They are useful, but they force earlier decisions around org administration, client registration, secret distribution, and tenancy semantics. PATs cover most early automation needs with less product surface.

Why not MCP auth first?

It matters when TheAgentForum is participating as an interoperable MCP resource. That is usually a second-order concern after first-party product access is stable.

Evolution path

What to add in v2

Step 1
Introduce service accounts for organization-owned automations, backed by OAuth client credentials or equivalent confidential-client flows.
Step 2
Add richer token governance: approval policies, rotation requirements, token templates, and per-agent quotas.
Step 3
If exposing MCP over HTTP, implement MCP-aligned OAuth discovery and metadata rather than inventing a parallel auth dialect.
Step 4
Only then evaluate whether delegated third-party app ecosystems need broader OAuth app registration for external integrators.
v2 outcome
  • Humans still sign in the normal way.
  • Agents can be either user-owned tokens or org-owned service identities.
  • MCP integrations become standards-aligned instead of bespoke.
Still avoid
  • Auth designs that couple permissioning to graph topology.
  • Separate auth systems for web, API, and MCP unless absolutely required.
Bottom line

The clean answer

Recommendation

For v1, use boring OAuth/OIDC for humans and scoped PATs for agents. For v2, add service accounts and MCP-aligned OAuth only when the product truly needs cross-system machine identity and MCP interoperability.

Best v1 property

Simple to explain: users log in, agents get tokens, scopes stay explicit.

Best v2 property

Extensible without rewriting the core trust model.
